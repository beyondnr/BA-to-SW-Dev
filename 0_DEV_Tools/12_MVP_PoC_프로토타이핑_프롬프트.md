## Firebase Studio용 MVP PoC 프론트엔드 프로토타이핑 프롬프트

아래 프롬프트를 그대로 Firebase Studio / 코드 생성 에이전트에 입력하여, PRD 및 Epic 0 FE Task 정의에 맞는 프론트엔드 PoC를 생성하도록 지시한다.

---

### 전체 프롬프트 (Firebase Studio에 그대로 붙여넣기)

```text
당신은 **SaaS형 직원 스케줄 관리 서비스(LawfulShift 가칭)**의 MVP PoC를 구현하는
시니어 프론트엔드 엔지니어이자 UX 디자이너입니다.
React + TypeScript + Vite + TailwindCSS 스택을 사용해, 하나의 SPA 형태로
핵심 사용자 플로우를 "넓고 얇게" 프론트엔드 프로토타이핑하는 것이 목표입니다.

아래 PRD 요약과 Epic 0 프론트엔드 Task 정의(EPIC0-FE-001 ~ 004)를 기준으로
Firebase 프로젝트 안에 동작 가능한 프론트엔드 PoC를 만들어 주세요.
백엔드 API는 실제로 연동하지 않고, **Mock 데이터/Mock API 호출**로 대체합니다.

--------------------------------
1. 제품/PRD 요약 (기준 컨텍스트)
--------------------------------
- 제품 이름(가칭): LawfulShift – 소상공인용 직원 스케줄 & 급여 관리 SaaS
- 핵심 타겟: 디지털 기기에 익숙하지 않은 소상공인 사장님 (베이커리, 국밥집, 꽃집 등)
- 핵심 문제:
  - 수당/급여 계산 오류 → 과태료/임금체불 위험
  - 직원 스케줄 관리가 카톡/전화/수기로 분산 → 승인까지 오래 걸림
  - 분쟁 발생 시 증빙/기록이 부족
- 핵심 목표:
  - 법규 준수 자동화 (노동법 규칙 엔진 기반 경고)
  - 스케줄 관리 간소화 (3단계 온보딩, 1-Click 승인)
  - 증빙 자료 자동 생성/보관 (PDF/엑셀, Audit Log)
- MVP v1에서 프론트엔드 PoC로 확인하고 싶은 흐름:
  1) 신규 사장님 온보딩 (가입 → 매장 정보 → 직원 등록)
  2) 직원 가용시간 제출(모바일)
  3) 사장님 스케줄 대시보드에서 검토/수정/승인
  4) 승인된 스케줄 기준 급여 미리보기 & 리포트 확인

이번 PoC의 목적은 **위 네 가지 흐름이 화면 상에서 자연스럽게 이어지는지**,
그리고 PRD의 “3단계 온보딩 + 1클릭 승인 + 급여 미리보기” 경험을
시각적으로/인터랙션 수준에서 검증하는 것입니다.

--------------------------------
2. 기술 스택 및 공통 구현 원칙
--------------------------------
[필수]
- React 18 이상 + TypeScript
- Vite 기반 프로젝트 구조
- TailwindCSS로 스타일링 (모바일 퍼스트, 데스크탑 대응은 기본 수준)
- React Router를 활용한 SPA 라우팅

[공통 UI/UX 원칙]
- PRD의 페르소나(디지털에 익숙하지 않은 사장님/직원)를 고려해,
  최대한 단순한 화면 구조와 큰 버튼, 명확한 문구를 사용합니다.
- 모든 주요 액션에는 명확한 피드백(토스트, 라벨, 간단한 메시지)을 제공합니다.
- PoC 수준이므로 **픽셀 퍼펙트, 복잡한 애니메이션, 고급 접근성**은 Out of Scope입니다.
- 실제 백엔드 호출 대신 `setTimeout`과 Mock 데이터로 응답을 시뮬레이션합니다.

[폴더 구조 예시]
- `src/main.tsx` : 엔트리
- `src/App.tsx` : 전체 라우팅 및 레이아웃
- `src/pages/Onboarding/*` : 온보딩 3단계 화면
- `src/pages/AvailabilityForm/*` : 직원 가용시간 제출 폼
- `src/pages/Dashboard/*` : 사장님 스케줄 대시보드 & 승인 화면
- `src/pages/Reports/*` : 급여 미리보기 & 리포트 화면
- `src/components/common/*` : Button, Input, Select 등 공통 컴포넌트
- `src/mock/*` : Mock 데이터 및 유틸리티

3. EPIC0-FE-001: 온보딩 및 매장/직원 관리 PoC UI
--------------------------------
목표:
- 신규 사장님을 위한 3단계 온보딩(가입 → 매장 → 직원) 위자드 UI와
  온보딩 이후 기본 설정을 확인/수정할 수 있는 간단한 관리 화면을 구현합니다.

[라우트]
- `/` 또는 `/landing` : 랜딩/시작하기
- `/onboarding` : 온보딩 위자드 진입
- `/settings` : 온보딩 완료 후 매장/직원 정보 확인/수정 페이지 (간단 버전)

[화면 및 기능 요구사항]
1) Landing/Login (간단)
   - 브랜드/서비스 한 줄 설명, "시작하기" 버튼.
   - "시작하기" 클릭 시 `/onboarding`으로 이동.

2) Onboarding Wizard (3단계)
   - 상단에 Step Indicator (1/3, 2/3, 3/3) 또는 Progress Bar.
   - Step 1: 사장님 계정 정보
     - 필드: 이메일, 비밀번호
     - 간단한 유효성 검사: 이메일 형식, 비밀번호 최소 길이 등
   - Step 2: 매장 정보
     - 필드: 매장명, 업종(Select), 영업시간(간단한 텍스트 또는 Time Range 필드)
   - Step 3: 초기 직원 등록
     - 최소 1명 이상의 직원 정보를 입력할 수 있는 리스트 UI
     - 필드: 이름, 시급(숫자), 역할(Select: 직원/매니저 등)
     - "직원 추가" 버튼으로 Row를 동적으로 추가/삭제 가능
   - 각 단계 하단:
     - "이전" / "다음" / "완료" 버튼
     - "다음" 클릭 시 필수값 유효성 검사 후에만 다음 단계로 이동
   - 위자드는 Context/전역 상태(Zustand, Context API 등 아무거나)로
     3단계 모두에서 입력된 값을 유지합니다.

3) 완료 시 동작
   - "완료" 버튼 클릭 시:
     - 콘솔에 전체 온보딩 데이터(JSON) 출력
     - 간단한 성공 메시지(토스트/알럿) 표시
     - `/dashboard` 또는 `/settings`로 네비게이션 (Mock 대시보드 진입)

4) Settings Page (간단 버전)
   - 온보딩에서 입력한 매장 정보, 직원 목록을 카드/테이블 형태로 보여줍니다.
   - 당장은 편집 기능이 필수가 아니며, PoC 수준으로는 "읽기 전용" 또는
     간단한 인라인 수정 정도만 제공합니다.

4. EPIC0-FE-002: 직원 가용시간 제출 모바일 폼 PoC UI
--------------------------------
목표:
- 직원이 **로그인 없이 링크/QR 코드**로 접속해,
  자신의 요일별 근무 가능 시간을 모바일에서 입력/제출하는 경험을 PoC합니다.

[라우트]
- `/availability` : 직원 가용시간 제출 폼

[화면 및 기능 요구사항]
1) Intro 화면
   - 상단에 안내 문구:
     - 예: "OOO 매장 3월 2주차 근무 가능 시간을 입력해주세요."
   - 현재 주차/기간은 Mock 텍스트로 표시.

2) Time Picker UI
   - 모바일 퍼스트 레이아웃 (375px 기준)
   - 요일 목록(Mon~Sun)을 탭 또는 아코디언 형태로 나열.
   - 각 요일마다 "시간 범위"를 하나 이상 추가할 수 있는 UI:
     - 예: "월요일 09:00 ~ 14:00"
     - "+ 시간 추가" 버튼으로 여러 구간 추가 가능.
   - 시간 입력은 다음 중 하나로 구현:
     - HTML5 `<input type="time">`
     - 또는 간단한 Select Box (예: 09:00, 09:30, 10:00 ...)

3) 유효성 검사
   - Start < End인지 기본 체크.
   - 최소 한 개 이상의 시간 구간이 있어야 "제출" 버튼 활성화.
   - 잘못된 경우, 간단한 에러 메시지(빨간 텍스트) 표시.

4) 제출 & 완료
   - "제출하기" 버튼 클릭 시:
     - 선택된 데이터를 `[ { day, start, end }, ... ]` 형태의 JSON으로 콘솔에 출력.
     - `setTimeout`을 사용해 1~2초 후 성공 응답을 시뮬레이션.
     - 성공 후 "제출되었습니다" 화면으로 전환 (동일 라우트 내 상태 전환이거나
       `/availability/success` 서브 라우트 등 자유롭게 선택).
   - 성공 화면에는 큰 체크 아이콘과 간단한 문구를 표시:
     - 예: "가용시간이 제출되었습니다. 감사합니다!"

5. EPIC0-FE-003: 사장님 스케줄 대시보드 및 승인 PoC UI
--------------------------------
목표:
- 사장님이 직원 가용시간 기반으로 생성된 "주간 스케줄"을 직관적으로 확인하고,
  간단한 수정과 1-Click 승인 경험을 PoC합니다.

[라우트]
- `/dashboard` : 메인 스케줄 대시보드

[화면 및 기능 요구사항]
1) Dashboard Main
   - 상단 요약 카드:
     - "승인 대기 스케줄 1건", "이번 주 예상 인건비: XXX,XXX원" 등 Mock 데이터로 표시.
   - "승인 대기" 상태를 강조하는 카드 UI (3단계: 검토 → 조정 → 공지 라벨 정도).

2) Schedule View (주간 스케줄 테이블/캘린더)
   - 가로축: 날짜/요일 (예: 월~일)
   - 세로축: 직원 이름
   - 그리드 셀 내부에 근무 시간 표시 (예: "09:00~14:00")
   - 가용시간 미제출 직원은 흐리게 처리하거나 별도 아이콘으로 구분.
   - CSS Grid 또는 적절한 테이블 레이아웃 사용 (반응형 고려).

3) Shift 수정 인터랙션
   - 셀 클릭 시 간단한 Popover/Modal을 띄워
     - 시작/종료 시간을 수정할 수 있게 합니다 (입력 방식은 AvailabilityForm과 유사).
   - "저장" 클릭 시 로컬 상태만 업데이트 (백엔드 호출 없음).

4) 승인 버튼 & 규칙 위반 시뮬레이션
   - 화면 상단 또는 하단 Action Bar:
     - "스케줄 승인", "직원에게 알림 발송" 버튼 배치.
   - "스케줄 승인" 클릭 시:
     - Random Success/Fail 로직으로 규칙 위반을 시뮬레이션.
     - Fail인 경우:
       - 주 52시간 초과, 휴게시간 부족 등 예시 문구를 포함한
         빨간색 토스트/모달을 띄웁니다.
       - 위반된 직원/요일을 간단히 하이라이트할 수 있으면 더 좋습니다(옵션).
     - Success인 경우:
       - "승인 완료" 뱃지/라벨을 상단 카드에 표시.
       - "직원에게 알림이 전송되었습니다(모의)" 토스트 표시.

6. EPIC0-FE-004: 급여 미리보기 및 리포트 PoC UI
--------------------------------
목표:
- 승인된 스케줄을 기반으로 계산된 **예상 급여 명세**와
  간단한 리포트/증빙 다운로드 UI를 PoC합니다.

[라우트]
- `/reports` : 급여 미리보기 & 리포트 화면

[화면 및 기능 요구사항]
1) Report Header
   - 조회 기간 선택(예: 주차 선택 드롭다운 또는 단순 텍스트).
   - "총 예상 인건비: XXX,XXX원" 표시.

2) Payroll Table
   - Row: 직원 이름
   - Columns 예시:
     - 총 근무시간, 시급, 기본급, 주휴수당, 연장수당, 야간수당, 휴일수당, 총 지급액
   - 각 셀의 금액은 Mock 데이터로 계산된 값처럼 보이게 합니다.
   - 금액은 3자리 콤마 포맷팅(예: 1,200,000) 유틸리티 함수로 출력.
   - Row 클릭 시 일별 근무 기록을 토글로 펼쳐서 보여주는
     간단한 Expandable Row 구현 (옵션이지만 가능하면 포함).
   - 전월/전주 대비 변동이 있는 경우(예: 급여 증가)는
     녹색/빨간색 하이라이트나 뱃지로 표시합니다 (Mock 기준).

3) Export / Print Controls
   - "Excel 다운로드", "PDF 다운로드", "인쇄" 버튼을 상단/하단에 배치.
   - PoC이므로:
     - "Excel/PDF 다운로드"는 실제 파일 대신 토스트/알럿으로
       `"Mock 다운로드가 시작되었습니다"` 정도의 메시지만 출력.
   - CSS `@media print` 또는 Tailwind의 `print:*` 유틸을 활용하여
     인쇄 시에는 네비게이션 바/불필요한 UI를 숨기고
     테이블 본문만 깔끔하게 출력되도록 간단한 스타일을 추가합니다.

7. 상태 관리, Mock 데이터, 코드 품질 요구사항
--------------------------------
- 상태 관리:
  - 간단한 범위에서는 React useState/useContext로 처리해도 좋고,
    필요하다면 경량 상태 관리 라이브러리(Zustand 등)를 사용해도 됩니다.
  - 온보딩에서 입력한 매장/직원 정보는 Dashboard/Reports에서
    일부라도 재사용할 수 있게 구조화해 주세요 (예: 직원 목록).

- Mock 데이터:
  - `src/mock` 디렉터리에 직원/스케줄/급여 데이터를 정의합니다.
  - Dashboard와 Reports는 동일한 Mock 스케줄/급여 데이터를 공유해도 됩니다.

- 코드 품질:
  - TypeScript 타입을 명확히 정의하고, any 사용은 최소화합니다.
  - 컴포넌트는 적절히 분리하여 재사용성을 높이고,
    지나치게 거대한 단일 파일은 피합니다.
  - Tailwind 클래스는 가능한 한 읽기 쉽게 정리합니다.

8. 네비게이션 흐름 (엔드 투 엔드 시나리오)
--------------------------------
아래 플로우를 브라우저에서 자연스럽게 따라갈 수 있어야 합니다.

1) `/` 또는 `/landing` 에서 "시작하기" 클릭 → `/onboarding`
2) 온보딩 3단계를 모두 완료 → `/dashboard` 또는 `/settings` 진입
3) (별도 탭/창에서) 직원이 `/availability`에 접속하여 가용시간 제출
4) 사장님이 `/dashboard`에서 스케줄을 확인/간단 수정 후 "스케줄 승인"
5) 승인 완료 후 `/reports`에서 급여 미리보기 & 인쇄/다운로드 버튼 확인
```
